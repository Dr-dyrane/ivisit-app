## 3. WHY THESE VIEW HOOKS WERE MISSED

### Reason 1: Location Blind Spot
- **Pattern**: Hooks in `components/map/views/` were treated as "component internals"
- **Reality**: These are 200-860 line state machines that just happen to live in components/
- **Analogy**: Like finding a database inside a React component

### Reason 2: False Completion Signal
- `useMapExploreFlow` migration appeared complete
- But view controllers still have embedded complexity
- MapScreen.jsx deprecation notice mentions decomposition but views weren't prioritized

### Reason 3: Payment State Gap
The **most critical** missed opportunity is `useMapCommitPaymentController`:
- 860 lines of payment transaction state
- Manages: `selectedPaymentMethod`, `estimatedCost`, `isSubmitting`, `submissionState`, etc.
- Uses `paymentService` directly (no TanStack Query)
- **This is exactly the kind of server-interactive state TanStack Query was designed for**

## 4. HIGH-VALUE MIGRATION TARGETS

### 4.1 Priority 1: useMapCommitPaymentController -> Jotai + TanStack Query

**Current State:**
```javascript
// 11 useState calls, 860 lines

const [estimatedCost, setEstimatedCost] = useState(...);

```

**Proposed Atoms:**
```javascript
// atoms/paymentAtoms.js
export const selectedPaymentMethodAtom = atom(null);
export const paymentMethodsListAtom = atom([]);
export const estimatedCostAtom = atom(null);
export const paymentSubmissionStateAtom = atom('idle'); // 'idle' | 'submitting' | 'processing' | 'completed'
export const paymentErrorAtom = atom(null);
```

**Proposed TanStack Query Hooks:**
```javascript
// hooks/payment/usePaymentMethodsQuery.ts
export function usePaymentMethodsQuery() {
  return useQuery({
    queryKey: ['paymentMethods'],
    queryFn: () => paymentService.getPaymentMethods(),
    staleTime: 5 * 60 * 1000, // 5 minutes
  });
}

// hooks/payment/usePaymentIntentMutation.ts
export function usePaymentIntentMutation() {
  return useMutation({
    mutationFn: paymentService.createEmergencyCardPaymentIntent,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['activeTrip'] });
    },
  });
}
```

**Benefits:**
- Transaction state becomes atomic and debuggable
- Automatic caching of payment methods
- Optimistic updates for submission state
- Eliminates 400+ lines of manual state management

### The Pattern Problem
```
WRONG MENTAL MODEL:

  -> Must only contain presentational logic
  -> State belongs in hooks/ directory

ACTUAL REALITY:
components/map/views/ = "Feature Modules"
  -> Each contains a controller hook (200-860 lines)
  -> These ARE business logic, just misplaced
  -> Perfect candidates for Jotai atoms
```

### Directory Structure Confusion
The project has:
- `hooks/map/exploreFlow/` - 11 hooks - Migrated
- `hooks/map/screen/` - 4 hooks - Created
- `hooks/map/shell/` - 3 hooks You created
- `components/map/views/*/useMap*Controller.js` - 5 hooks ! MISSED

The view controllers follow a pattern: `useMap[Feature]Controller`  
They're hooks in disguise, just inside components/.
