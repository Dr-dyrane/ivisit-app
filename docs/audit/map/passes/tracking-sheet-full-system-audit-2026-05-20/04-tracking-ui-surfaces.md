### Induction Proof - Live Chrome Session

Invariant under test: live tracking chrome may be optimistic immediately after a
commit, but after that optimism settles it must be governed by lifecycle plus
active request identity, not by request key alone. Tracking readiness is a
separate, stronger predicate.

| State / transition             | Source step                                                                                                                                                        | Invariant result                                                                                                                                | Proof verdict                                 |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------- |
| idle / no active request       | `useMapTracking()` closes `TRACKING` when `trackingRequestKey` is falsy.                                                                                           | No key means no tracking sheet.                                                                                                                 | Proved for missing-key base case.             |
| pending / active / arrived     | `useTripLifecycle()` derives `hasActiveTrip` from pending approval, active, arrived, and completing, excluding completed/cancelled/rating-pending.                 | These states are eligible for live chrome, but not necessarily tracking-ready.                                                                  | Proved as lifecycle predicate.                |
| payment commit to tracking     | `useMapCommitFlow.finishCommitPayment()` clears commit flow and calls `openTracking()` directly; `useMapTracking()` also force-opens when coming from commit.      | Immediate sheet opening is intentional optimism and may occur before full route/responder/hospital readiness.                                   | Proved as intentional exception.              |
| direct `openTracking()` call   | `openTracking()` resolves hospital context and sets a tracking sheet view; it does not itself check `hasActiveTrip`.                                               | Any caller that invokes `openTracking()` must already prove lifecycle/request identity or accept commit optimism.                               | Not globally proved; caller-specific burden.  |
| lifecycle false with stale key | `useMapTracking()` auto-close checks only `!trackingRequestKey`; `useMapTrackingHeader()` visibility also checks key/phase/snap/modal, not `hasActiveTrip`.        | A stale key can keep or resurrect live chrome after lifecycle turns completed/cancelled/rating-pending.                                         | Finding survives induction.                   |
| completed/rating moment        | `tripLifecycleMachine` sets completed/rating context, while `hasActiveTrip` excludes completed; header presentation can still say `Complete` if visible.           | Terminal/rating ownership is correct in lifecycle, but live chrome suppression is not fully proven because visibility is not lifecycle-derived. | Lifecycle proved; chrome consumer incomplete. |
| tracking-ready flag            | `buildTrackingRuntimeSnapshot()` computes `isTrackingReady` from request id plus stage metadata; stage metadata marks assigning/pending/lost/delayed as not ready. | Snapshot readiness is useful for copy/action honesty, but it is not the same as the lifecycle predicate that should hide/show live chrome.      | Boundary proved.                              |

Induction conclusion: the code proves the lifecycle predicate exists and that
commit optimism is intentional. It does not prove live chrome is lifecycle-gated
for every transition because the close/header consumers still accept
`trackingRequestKey` without `hasActiveTrip`. The target fix remains a named
session predicate such as `isTrackingSessionActive =
Boolean(trackingRequestKey) && hasActiveTrip`, while preserving the separate
commit optimism and tracking-readiness predicates.

### Induction Proof - Pending And No-Responder Semantics

Invariant under test: a request may be validly pending or active before canonical
responder assignment arrives. In that window, visible tracking surfaces must not
claim live movement or a confirmed driver unless the source fields prove that
meaning.

| State / transition                    | Source step                                                                                                                                                                      | Invariant result                                                                                                                                                    | Proof verdict                                       |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| pending approval                      | `resolveTrackingStage()` returns `pending_approval` for pending kind/status before movement or responder checks; hero says `Awaiting approval`; sheet header says `Confirming`.  | Pending sheet/hero semantics do not claim dispatch or movement.                                                                                                     | Proved for sheet/hero.                              |
| pending floating header               | `buildMapActiveSessionHeaderSession()` returns `Awaiting approval` for cash pending and `Processing` for other pending requests.                                                 | Header avoids `En Route` while `activeMapRequest.kind` is pending.                                                                                                  | Proved for pending header.                          |
| approval before responder fields      | `EmergencyRequestModal` explicitly allows approved ambulance cash flow without responder assignment, shows assigning-driver copy, and passes `assignedAmbulance: null`.          | A valid approved request can enter tracking before assignment fields exist.                                                                                         | Proved as valid state, not error.                   |
| request completion handoff            | `handleRequestComplete()` starts ambulance tracking with accepted status, route/ETA fallback, and only explicit `assignedAmbulance` / responder fields from the request payload. | The request handoff can have ETA/route but no canonical responder assignment.                                                                                       | Proved as possible state.                           |
| local start fallback                  | `startAmbulanceTrip()` fills `assignedAmbulance` from `activeAmbulances` by id, hospital, available ambulance, or first ambulance when explicit assignment is absent.            | Runtime `hasResponder` / `responderName` can become truthy from local ambulance inventory, not only from backend assignment.                                        | Finding survives; responder truth is non-canonical. |
| query hydration partial responder     | `buildAmbulanceTripSnapshot()` treats responder fields, ambulance id, fetched ambulance, or responder location as responder identity and preserves previous assigned ambulance.  | Hydration preserves useful runtime identity, but `hasResponder=true` may mean "runtime responder-like data exists," not "assignment proof reached payment handoff." | Proved as nuanced runtime truth.                    |
| active ambulance, no responder/no ETA | `resolveTrackingStage()` returns `assigning` for accepted/in-progress ambulance when there is no responder and no movement signal; hero says `Assigning driver`.                 | Sheet and hero keep assigning/preparing semantics for the clean no-responder/no-route base case.                                                                    | Proved for sheet/hero.                              |
| active ambulance, no responder + ETA  | `resolveTrackingStage()` returns `dispatch_confirmed` for active status plus route/ETA even without responder.                                                                   | Route/ETA is treated as movement/logistics signal and can promote copy before responder assignment.                                                                 | Product decision remains open.                      |
| floating header for active ambulance  | `resolveSessionStatusLabel()` returns `En Route` for any ambulance that is not pending, arrived/complete, or telemetry lost/stale.                                               | Header skips `assigning` and `dispatch_confirmed`, so it can overstate no-responder active states that the sheet represents more carefully.                         | Finding survives induction.                         |

Induction conclusion: pending state is protected, and the sheet/hero can
represent the clean no-responder/no-route state honestly. The proof breaks at
two boundaries: route/ETA can promote active no-responder trips to
`dispatch_confirmed`, and the floating header collapses almost every active
ambulance state to `En Route`. Separately, local ambulance fallback means
`hasResponder` is runtime enrichment, not canonical backend assignment proof.

### Induction Proof - Route / ETA Request Ownership

Invariant under test: route geometry, ETA, distance, and derived progress must
belong to the same active request and destination hospital that the tracking
sheet is describing. A route callback should never make a stale hospital,
ambient location, or previous request look like the current trip's route truth.

| State / transition               | Source step                                                                                                                                                                                                              | Invariant result                                                                                                                                        | Proof verdict                                     |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| route calculation key            | `useMapRoute()` builds route cache keys from origin/destination coordinates only, not request id or hospital id.                                                                                                         | Route cache is spatial truth, not request truth. It can safely reuse identical coordinates, but cannot by itself prove request ownership.               | Proved as coordinate-scoped only.                 |
| preview map route producer       | `EmergencyLocationPreviewMap` calculates route from `routeOriginCoordinate` (`location`) to `routeDestinationCoordinate` (`focusedCoordinate` or selected hospital).                                                     | The route producer follows ambient map origin and selected/focused destination, not committed request pickup/hospital identity directly.                | Ownership depends on upstream focus correctness.  |
| route callback payload           | `EmergencyLocationPreviewMap` emits duration, distance, coordinates, and calculating state through `onRouteInfoChange`; it does not emit request id or hospital id.                                                      | The callback is unowned route data until another layer stamps it.                                                                                       | Proven unscoped producer boundary.                |
| callback stamping                | `useMapTrackingSync.setTrackingRouteInfo()` overwrites incoming route payloads with `requestKey: normalizedActiveRequestKey` and `routeSource: "live_route"`.                                                            | The atom is request-scoped at receipt time, which prevents unkeyed route data from being read by unrelated requests later.                              | Proved as contextual stamping.                    |
| request key change               | `useMapTrackingSync()` resets calculation/error state and reseeds `trackingRouteInfoAtom` whenever `normalizedActiveRequestKey` changes.                                                                                 | A new request does not blindly keep the previous request key; route atom scope changes with active model identity.                                      | Proved for key transition.                        |
| ambulance store seeding          | On matching ambulance key, `useMapTrackingSync()` seeds route atom from `activeAmbulanceTrip.etaSeconds` and `activeAmbulanceTrip.route`, preserving same-key duration when needed.                                      | Reload/hydration can restore useful same-request ETA/route before a new map callback fires.                                                             | Proved for same-key ambulance.                    |
| route atom read in tracking      | `useMapTrackingRuntime()` only accepts `trackingRouteInfoAtom` when `liveRouteInfo.requestKey === trackingRouteRequestKey`.                                                                                              | Tracking runtime will not consume a route atom stamped to a different active key.                                                                       | Proved for route atom consumer.                   |
| store patch from route           | `useMapTrackingSync()` patches `activeAmbulanceTrip` only when tracking sheet is active, kind is ambulance, trip has a request id, and route/timeline drifts.                                                            | Store mutation is limited to active ambulance tracking, but it trusts the already-stamped route payload.                                                | Guarded, but inherits producer correctness.       |
| hospital/destination mismatch    | `EmergencyLocationPreviewMap` has `hasRouteTargetMismatch` guards and can reuse canonical tracking route when active tracking/timeline exists.                                                                           | The map has some spatial sanity checks, but route payload still lacks the active request's hospital id, so wrong focus can still be stamped as current. | Finding survives as contextual-stale route risk.  |
| active request versus map focus  | `MapScreen` passes `selectedHospitalId={mapFocusedHospitalId}` and route callbacks to the same map, while `useMapFocusedState()` can prioritize history or shell focus before active request fallbacks in some branches. | Route destination ownership depends on focus resolver correctness; route atom cannot independently verify the hospital belongs to the active request.   | Finding survives; needs active hospital resolver. |
| coordinate drift / pickup source | `EmergencyLocationPreviewMap` route origin follows live `activeLocation`; request `patientLocation` is preserved elsewhere on trip/query state.                                                                          | Route ETA can be current-location scoped even when the committed pickup coordinate is different.                                                        | Finding survives as pickup-origin split.          |

Induction conclusion: route/ETA ownership is strong after data enters
`trackingRouteInfoAtom` because reads and seeding are request-key scoped. The
proof does not close at the producer boundary: the emitted route payload has no
request id or hospital id and is stamped contextually with the current active
key. Therefore a stale map focus, ambient origin change, or active hospital
resolver mistake can still attach plausible route geometry to the current
request. The code defends against cross-request atom reads; it does not prove
route geometry belongs to the current request's committed pickup and hospital.

### Induction Proof - Map Render Lifecycle

Invariant under test: map rendering may animate, refit, redraw, or recreate
visual objects, but those visual lifecycle events must not be confused with
backend/request truth. The only render-to-state bridge that can affect tracking
truth is the route-info callback path already covered by the route ownership
proof.

| Render event / state           | Source step                                                                                                                                                                 | Invariant result                                                                                                                                      | Proof verdict                   |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| web marker creation            | `MapComponents.web.js` creates a Google marker in an effect and removes it with `setMap(null)` on dependency changes.                                                       | Marker teardown/recreate is a render lifecycle event; it does not mutate active trip, lifecycle, route atom, or backend by itself.                    | Proved as visual churn.         |
| web marker dependency churn    | Marker dependencies include `coordinate`, `anchor`, `centerOffset`, `image`, `imageSize`, `onPress`, `title`, `opacity`, and selection fields.                              | Ordinary React object/function identity changes can recreate markers even when tracking state is valid.                                               | Proved flicker cause.           |
| hospital marker props          | `HospitalMarkers` passes inline `onPress`, inline `anchor`, fresh `centerOffset`, and hospital coordinate objects into the marker.                                          | Hospital marker flicker on web can come from prop identity/render churn, not hospital assignment or browser geolocation truth.                        | Proved visual-source issue.     |
| ambulance sprite bucket        | `RouteLayer` keys ambulance marker by `ambulance-${ambulanceSpriteBucket}` and swaps sprite images as heading changes.                                                      | Heading bucket changes intentionally remount the ambulance marker, so sprite flicker can happen without responder identity changing.                  | Proved sprite remount source.   |
| ambulance coordinate animation | `EmergencyLocationPreviewMap` and `RouteLayer` feed animated/service marker coordinates into markers during active tracking or timeline reuse.                              | Continuous ambulance movement is expected; on web it becomes marker recreate churn because marker position changes are dependency changes.            | Proved motion-to-render churn.  |
| polyline lifecycle             | `MapComponents.web.js` `Polyline` creates a Google polyline in an effect and removes it on dependency changes, including the rest `props` object.                           | Route line redraw can happen on render even when route ownership and ETA are valid.                                                                   | Proved visual churn.            |
| marker render pulse            | `EmergencyLocationPreviewMap` builds `markerRenderPulseKey` from visible hospitals, selected/user/service coordinates, and ambulance sprite bucket.                         | Marker view refreshing is intentionally keyed to coordinate/sprite changes; this is presentation stabilization, not request-state proof.              | Proved presentation-only key.   |
| camera route fit               | `EmergencyLocationPreviewMap` fits route coordinates when `routeFitPrimeKey` changes, and refits on occlusion changes or web pan compensation.                              | Camera movement can make tracking appear unstable, but it does not mutate active trip or lifecycle state.                                             | Proved visual-only movement.    |

| Chrome/geolocation suspicion   | `useMapLocation()` does not start continuous watch tracking on web, while map marker/polyline effects can churn without location watcher updates.                           | Chrome location can be an input or amplifier, but source proof points to app-owned marker/polyline/camera lifecycle as the primary web flicker class. | Location-only theory disproved. |

Induction conclusion: map render churn is now source-proved as a separate
presentation class. Marker, sprite, polyline, and camera lifecycles can flicker
or continuously redraw while backend/store tracking state remains valid. The
only visual path that crosses into tracking truth is route-info emission, which
is already classified as context-stamped and request-key scoped but not
hospital/pickup-proven. Therefore marker flicker should not be treated as proof
of bad Chrome geolocation or backend tracking failure.

### Route Request-Key Parity Detail

| Join point                | Current implementation evidence                                                                                                                                                 | Audit interpretation                                                                                                                                 |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Active key source         | `useMapExploreFlow()` sets `trackingRequestKey = activeMapRequest.requestId`.                                                                                                   | The entire route/ETA scope depends on the active model choosing the same request id shape as the active trip.                                        |
| Route writer              | `useMapTrackingSync.setTrackingRouteInfo()` overwrites incoming route payloads with `requestKey: normalizedActiveRequestKey`.                                                   | Map route callbacks do not carry their own identity; the sync hook stamps the current active key at receipt time.                                    |
| New request reset         | When `normalizedActiveRequestKey` changes, `useMapTrackingSync()` resets route error/calculation state and seeds from store ETA/route only if ambulance key matches active key. | Correctly avoids cross-request leakage, but a mismatched key prevents otherwise useful stored ETA/route from seeding the atom.                       |
| Same-request preservation | During calculation, `mergeTrackingRouteInfo()` preserves existing duration, distance, and coordinates when incoming route data is incomplete.                                   | Good no-flash behavior only applies after the route atom is already scoped to the same request key.                                                  |
| Runtime read              | `useMapTrackingRuntime()` accepts `liveRouteInfo` only when `liveRouteInfo.requestKey === trackingRouteRequestKey`; otherwise it uses null fallback.                            | The sheet can drop ETA/progress display without losing the active trip itself.                                                                       |
| Map callback producer     | `FullScreenEmergencyMap` calls `onRouteCalculated()` with coordinates, duration, and distance, but no request id.                                                               | Route identity is entirely contextual; if map focus/hospital state is stale when the callback fires, route data can be stamped to the wrong context. |
| Derived atoms             | Generic route derived atoms read `trackingRouteInfoAtom` without checking active request key.                                                                                   | Generic route UI helpers should not be used as proof that active tracking route data belongs to the current request.                                 |

Fix shape: normalize a single canonical tracking request key at active trip creation and route sync, or carry an alias set through route reads. Runtime can then accept route data when it matches any active request alias while still preventing cross-request leakage.

### Pickup Truth Split

| Truth lane                   | Current implementation evidence                                                                                                                                                        | Audit interpretation                                                                                                                                                |
| ---------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Committed pickup coordinate  | Payment initiation builds `patientLocation` from commit-time `currentLocation`; `handleRequestComplete()` passes it into `startAmbulanceTrip()`; query hydration rebuilds it.          | The active request can retain a request-scoped pickup coordinate even after the map shell keeps evolving.                                                           |
| Commit pickup label handoff  | `buildAmbulanceCommitRequest()` and `buildBedCommitRequest()` capture `locationLabel`, but `buildCommitPaymentCompletionPayload()` forwards `patientLocation` and not `locationLabel`. | The commit lane has a human pickup label briefly, then tracking runtime loses that committed label unless another producer restores it.                             |
| Active tracking route origin | `MapScreen` passes `activeLocation` into `EmergencyLocationPreviewMap`; that map calculates `routeOriginCoordinate` from its `location` prop and recomputes the preview route.         | Fresh route callbacks are ambient-location scoped unless stored tracking route coordinates already carry the request-specific route back into the map.              |
| Tracking route card pickup   | `MapSheetOrchestrator` passes `currentLocationDetails` into tracking; `buildTrackingViewState()` builds `pickupLabel` and `pickupDetail` only from that `currentLocation`.             | Expanded route-card pickup copy is live shell display truth, not a rendered label recovered from trip `patientLocation`.                                            |
| Share payload pickup         | `buildTrackingSharePayload()` receives the same view-state `pickupLabel`.                                                                                                              | Shared ETA text can inherit ambient pickup-label drift even when the active trip's committed coordinate is still preserved.                                         |
| Map responder fallback       | Demo responder motion prefers `trip.patientLocation` and only falls back to current user location when the trip coordinate is absent.                                                  | The animation/runtime layer already treats request pickup coordinate as stronger than ambient current location in at least one downstream consumer.                 |
| Editing surface              | Tracking route card comments say pickup edits were removed because they changed local shell state rather than live request destination.                                                | The UI knows pickup edits during tracking were misleading; the remaining display path still uses the mutable shell location model for pickup copy and route origin. |

Audit reading: pickup is currently three truths, not one. `patientLocation`
anchors committed coordinate continuity, the preview map route origin follows the
ambient map location model, and the route card/share text follows the ambient
display label model. That split may be intentional for live-device tracking, but
it must be tested before audit closeout because it can produce a valid trip whose
displayed pickup text or newly calculated route no longer describes the request
that payment committed.

### Tracking Action Failure Feedback Evidence

| Action lane                    | Current implementation evidence                                                                                                                                      | Audit interpretation                                                                                                           |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| Shared busy wrapper            | `runBusyAction()` sets `busyAction`, awaits the handler, returns its result, and clears `busyAction` in `finally`.                                                   | It gives immediate loading feedback, but it does not inspect `{ ok: false }` or caught domain errors for user-facing copy.     |
