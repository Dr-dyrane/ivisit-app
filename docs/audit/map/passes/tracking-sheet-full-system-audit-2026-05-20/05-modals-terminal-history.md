| Rating has one renderer                          | `components/map/MapModalOrchestrator.jsx` renders a single `ServiceRatingModal`.                                                                                                                                   | Double flashes should be investigated as state changes/remounts, not as two rating components rendering together.    |
| Tracking rating wins over recovered rating       | `MapModalOrchestrator` chooses `trackingRatingState` when visible, else `recoveredRatingState`.                                                                                                                    | In-flow completion should suppress recovered rating content while the tracking rating is visible.                    |

| Runtime validation can suppress stale rating     | `useTrackingRatingFlow()` returns `INITIAL_TRACKING_RATING_STATE` when the visible rating visit is missing or already rated in loaded visits.                                                                      | The returned state can hide the modal, but the atom is not immediately rewritten by that pure validation.            |
| Completion opens rating after backend completion | `useMapTrackingController.handleCompleteAmbulanceWithRating()` calls `onCompleteAmbulanceTrip({ deferCleanup: true })`, writes a recovery claim, clears recovered rating state, then sets tracking rating visible. | Tracking is intentionally kept long enough to preserve rating context, so lifecycle/header cleanup must be explicit. |
| Raw close only clears modal state                | `useTrackingRatingFlow.closeRating()` resets `trackingRatingStateAtom` but does not finalize the completed trip.                                                                                                   | This lower-level handler is not a terminal resolver by itself.                                                       |
| Shell close resolves through skip                | `ServiceRatingModal.handleShellClose()` blocks during pending work, dismisses keyboard first, then calls `handleSkip()`; web Escape also calls `handleSkip()`.                                                     | User-facing shell close currently means "skip rating", not passive dismiss.                                          |
| Rating skip/submit finalize active trip cleanup  | `skipRating()` and `submitRating()` reset rating state, call `finalizeCompletedTracking()`, then trigger after-resolution refresh hooks.                                                                           | The clean terminal path is skip/submit; shell close reaches that path through skip.                                  |
| Rating submit is idempotent server-side          | `resolveTrackingRatingSubmit()` uses `visitsService.updateRating()` with an idempotent rated-at guard and patches local visit cache.                                                                               | Duplicate submissions should become no-op/safe, but duplicate modal exposure still hurts UX.                         |

### Completion / Lifecycle Evidence

| Contract                                         | Current implementation evidence                                                                                                                                                                   | Interpretation                                                                                                                                                             |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Lifecycle machine excludes completed from active | `useTripLifecycle()` computes `hasActiveTrip` from pending, active, arrived, and completing states; completed and cancelled are false.                                                            | Completed/rating can intentionally have trip data in store while lifecycle says no active live trip.                                                                       |
| Completion can defer store cleanup for rating    | `useEmergencyHandlers.onCompleteAmbulanceTrip()` and bed completion set backend status complete and visit lifecycle rating-pending, then skip `stopAmbulanceTrip()` when `deferCleanup` is true.  | The terminal record remains in Zustand so rating has context. Visibility must be gated by lifecycle, not just store identity.                                              |
| Query preserves terminal store record            | `useActiveTripQuery()` preserves a store trip with terminal status when active query returns null.                                                                                                | This protects rating context but makes key-only header/sheet visibility unsafe after completion.                                                                           |
| Realtime clears terminal trip records            | `mergeEmergencyRealtimeTrip()` returns null for completed, cancelled, and payment-declined statuses; `useEmergencyRealtime()` writes that null into the active trip store when the event matches. | Realtime and query intentionally diverge: realtime can clear the trip immediately while query may preserve it for rating context. This is a rating flash/settle edge.      |
| Rating submit/skip performs final cleanup        | `useTrackingRatingFlow.finalizeCompletedTracking()` calls `stopAmbulanceTrip()` or `stopBedBooking()` after successful skip/submit.                                                               | Submit/skip is the explicit handoff from rating-pending to no live tracking state.                                                                                         |
| Raw close does not finalize cleanup              | `closeRating()` only clears `trackingRatingStateAtom`; `ServiceRatingModal` shell close and Escape call `handleSkip()` instead of calling raw close directly.                                     | The visible shell is safer than the primitive handler, but any future direct `onClose` caller must not bypass skip/submit semantics.                                       |
| `useMapTracking` comments overstate close gating | The hook comment says lifecycle prevents key-truthy races, but the close branch only closes when `trackingRequestKey` is missing; `hasActiveTrip` only gates auto-open.                           | This is a doc/code mismatch and a real target: live tracking chrome should close when lifecycle is no longer active unless rating UI owns an intentional terminal surface. |
| Floating header is key-gated                     | `useMapTrackingHeader()` sets `trackingHeaderVisible` from `Boolean(trackingRequestKey)` plus phase/modal/snap checks.                                                                            | Header can remain eligible while XState says completed/cancelled if request key is preserved for rating.                                                                   |

### Rating Modal Evidence

| Contract                                   | Current implementation evidence                                                                                                                                                                     | Interpretation                                                                                                                                         |
| ------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Rating renderer is lifted above tracking   | `useTrackingRatingFlow()` owns rating handlers at `MapScreen` level; `MapModalOrchestrator` renders one `ServiceRatingModal` outside `MapTrackingStageBase`.                                        | The previous unmount-on-sheet-change bug is structurally addressed; new flashes should be audited as atom/visit timing, not duplicate sheet renderers. |
| Tracking rating wins over recovered rating | `MapModalOrchestrator` chooses `trackingRatingState` when visible, otherwise `recoveredRatingState`; it passes one handler set to the single modal.                                                 | If both states become visible, tracking rating takes priority but recovered state may appear after tracking clears unless recovery suppression holds.  |
| Raw close is not completion cleanup        | `closeTrackingRating()` only resets `trackingRatingStateAtom`; skip/submit are the paths that call `finalizeCompletedTracking()` and stop the active trip/bed.                                      | The modal shell close currently calls skip first, but the raw handler remains unsafe as a standalone terminal action.                                  |
| Skip completes post-completion lifecycle   | `resolveTrackingRatingSkip()` writes visit lifecycle `POST_COMPLETION`, deletes recovery claim, clears atom, finalizes tracking cleanup, refreshes visits, and shows toast.                         | Skip is a real terminal resolution path, not a passive close.                                                                                          |
| Submit is idempotency-aware                | `resolveTrackingRatingSubmit()` deletes the recovery claim optimistically, writes rating through `visitsService.updateRating()`, patches local visit cache, and rolls claim back on server failure. | Duplicate submit/rerender recovery is guarded; runtime proof should verify the modal does not resurface after successful rating.                       |
| Modal itself has local reset and pending   | `ServiceRatingModal` resets local rating/comment/tip/pending state on `visible`, loads wallet balance, blocks close during pending, and calls `close()` after successful skip/submit.               | A brief flash can occur from visible-state handoff plus modal-local reset/load; this should be measured with tracking/recovered atom logs.             |
| Shell close has keyboard-first semantics   | When the keyboard is open, `handleShellClose()` dismisses keyboard and returns without skipping.                                                                                                    | First close intent can be a keyboard dismissal; second close intent resolves the rating through skip.                                                  |

### Contact Dispatch Evidence

| Contract                                   | Current implementation evidence                                                                                                                                                  | Interpretation                                                                                                                                                           |
| ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Chat modal state is UI-only                | `emergencyChatModalVisibleAtom` and `activeEmergencyChatRequestIdAtom` only store modal visibility and selected request id.                                                      | Chat atoms should not own tracking truth, ETA, active trip, or lifecycle.                                                                                                |
| Controller chooses chat request id         | `handleOpenContactDispatch()` uses the active ambulance request id first, then active bed booking request id, then opens the modal.                                              | The request id must stay canonical; a display id here would create room or realtime mismatches.                                                                          |
| Close clears chat state                    | `EmergencyContactDispatchModal.handleClose()` sets modal visible false, clears active request id, clears composer text, and calls `onClose`.                                     | Closing chat should not clear tracking request key, route atom, active trip, or ETA.                                                                                     |
| Room and messages are enabled by UI        | `useEmergencyChatRoom()` and `useEmergencyChatMessages()` are enabled only when the modal is visible and the request or room id exists.                                          | Opening can have a short room-creation phase; tracking UI should remain stable through that phase.                                                                       |
| Realtime is lifecycle-gated                | `useEmergencyChatRealtime()` subscribes only when visible, room id exists, and the modal lifecycle is ready; cleanup unsubscribes on disable, dependency change, or unmount.     | A `SUBSCRIBED` then `CLOSED` log can be normal on close or remount, but it is suspicious if tracking changes too.                                                        |
| Modal mirrors prop into atom               | `EmergencyContactDispatchModal` mirrors the `visible` prop into `emergencyChatModalVisibleAtom`, then renders from the atom-backed `modalVisible`.                               | There are two visible inputs; audit should confirm they do not briefly disagree during tracking sheet rerenders.                                                         |
| Room ensure writes query cache only        | `useEmergencyChatRoom()` calls `emergencyChatService.ensureRoomForRequest()` through a mutation and stores the response under `emergencyChatQueryKeys.roomByRequest(requestId)`. | Room creation cannot directly erase ETA or trip state; any ETA loss during chat open/close points to parent tracking remount, active request key, or route atom scoping. |
| Message fetches preserve previous data     | `useEmergencyChatMessages()` is enabled by room id, uses `placeholderData: previousData`, and only invalidates the room message key.                                             | Chat loading should not blank the tracking sheet; if the sheet changes, inspect modal/sheet orchestration instead of message fetching.                                   |
| Chat realtime patches message cache only   | `useEmergencyChatRealtime()` appends, replaces optimistic echoes, updates, deletes, or invalidates `emergencyChatQueryKeys.messages(roomId)` for the room.                       | Chat subscription `CLOSED` is not an active-trip query invalidation by itself.                                                                                           |
| Chat mutations are optimistic and scoped   | `useEmergencyChatMutations()` cancels/invalidate only the room messages key, rolls back on send error, and requests demo dispatch reply through the chat service.                | Message send can affect chat list order/loading, but should not mutate tracking lifecycle, route, or hospital labels.                                                    |
| Modal orchestrator renders chat separately | `MapModalOrchestrator` renders `EmergencyContactDispatchModal` from chat atoms alongside the single rating modal renderer.                                                       | Contact Dispatch and rating can coexist at the orchestrator level; visibility rules must prevent both from feeling like active tracking surfaces after terminal states.  |

### Modal Visibility Ownership Evidence

| Boundary / surface                 | Current implementation evidence                                                                                                                                     | Audit interpretation                                                                                                                                                 |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Modal orchestrator rating priority | `MapModalOrchestrator` chooses `trackingRatingState` when visible, otherwise `recoveredRatingState`, and passes one handler set into one `ServiceRatingModal`.      | Rating has a single renderer, but recovered rating can appear after tracking rating clears if recovery suppression is not still true.                                |
| Modal orchestrator chat source     | `MapModalOrchestrator` reads `emergencyChatModalVisibleAtom` and `activeEmergencyChatRequestIdAtom`, then renders `EmergencyContactDispatchModal`.                  | Chat visibility is atom-owned at the orchestrator boundary; the parent `onClose` does not own cleanup.                                                               |
| Chat prop-to-atom mirror           | `EmergencyContactDispatchModal` receives `visible`, writes it back to `emergencyChatModalVisibleAtom`, but renders `MapModalShell` from atom-backed `modalVisible`. | There is a one-render opportunity for prop/atom disagreement; normal on close, suspicious only if it changes tracking key, route atom, or header state.              |
| Chat hook enable source            | Room, messages, and realtime hooks are enabled from the `visible` prop plus request/room/lifecycle, while the shell itself renders from `modalVisible`.             | A stale prop/atom split can keep the shell visible while hooks are disabled, or start hooks before shell render catches up; runtime logs should capture both values. |
| Chat lifecycle machine             | `emergencyChatRoomMachine` owns idle, ensuring, loading, ready, sending, reconnecting, archived, and error states; `CLOSE` returns ready/error/archived to idle.    | Chat `CLOSED` realtime logs are expected on modal close/remount and should not be interpreted as emergency request lifecycle changes.                                |
| Explore modal suppression          | `useMapExploreFlow()` includes `emergencyChatModalVisible` in `hasActiveMapModal`, which hides the floating tracking header while chat is open.                     | Header disappearing while Contact Dispatch is open is intentional. The audit target is whether it reappears from stale key/lifecycle state after close.              |
| Shell modal suppression            | `useMapShell()` derives another `hasActiveMapModal` from profile/guest/care/history/payment/auth/recovered/loading state, but not Contact Dispatch.                 | FAB/layout/recovery gates may not share the same modal definition as tracking header suppression. This split should be intentional or consolidated.                  |
| Recovered rating suppression       | `useMapHistoryFlow()` suppresses recovered rating when `hasActiveMapModal` is true or `activeMapRequest.hasActiveRequest` is true.                                  | Because the shell modal gate excludes Contact Dispatch, active request truth is the main protection against recovered rating surfacing during chat.                  |

Fix shape: name the modal gates by responsibility, for example `hasTrackingHeaderBlockingModal`, `hasRecoveryBlockingModal`, and `hasLayoutBlockingModal`. If they are intentionally different, document the difference; if not, consolidate them so chat/rating/history cannot disagree about whether tracking chrome should be suppressed.

### Cancel And Cleanup Evidence

| ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Pending approval cancel                  | `handleCancelPendingRequest()` updates request status to cancelled, cancels the visit, updates visit lifecycle to cancelled, then clears pending approval.                                | Pending sheet closes, header hides, triage state clears or becomes irrelevant, no active trip.                                                  |
| Active ambulance cancel                  | `onCancelAmbulanceTrip()` calls `setRequestStatus(cancelled)`, `cancelVisit()`, writes visit lifecycle `CANCELLED`, then `stopAmbulanceTrip()` only when the request batch succeeds.      | Server row, active trip store, lifecycle, route atom, and header all clear for the same id.                                                     |

| Patient update forbids payment-declined  | `patient_update_emergency_request()` rejects `payment_declined`, checks owner identity, validates status transition, sets transition context, and stamps cancelled/completed timestamps.  | Patient confirm arrival, cancel, and complete are legal only when the backend transition graph allows them for that exact current status.       |

### History Lifecycle Selector Evidence

| Boundary / selector                 | Current implementation evidence                                                                                                                                      | Audit interpretation                                                                                                                                |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Emergency source inference          | `inferSourceKind()` treats a visit with `requestId` or `lifecycleState` as `emergency`; otherwise it is a scheduled visit.                                           | Emergency history behavior depends on visit rows carrying either request identity or lifecycle state.                                               |
| Request type inference              | `inferRequestType()` maps bed-like type/lifecycle `occupied` to `bed`, ambulance/emergency labels to `ambulance`, otherwise `visit`.                                 | Bed and ambulance history actions can diverge before the detail model ever renders.                                                                 |
| Lifecycle-to-history status mapping | `monitoring`, `arrived`, and `occupied` become `active`; `post_completion` and `rating_pending` become `rating_pending`; rated/cleared/completed become `completed`. | Visit lifecycle is deliberately coarser in history than live tracking; "active" history still requires active-request identity proof before resume. |
| Active group includes rating        | `ACTIVE_HISTORY_STATUSES` includes both `active` and `rating_pending`, so rating-pending visits group under `active_now`.                                            | Rating-pending history can look current without being a live tracking session; modal/rating copy must distinguish these states.                     |
| Primary action derivation           | `resolvePrimaryAction()` maps `rating_pending` to `rate_visit`; active emergency ambulance to `resume_tracking`; active emergency bed/pending to `resume_request`.   | The primary action label is lifecycle-derived, not proof that a live request still exists.                                                          |
| History item capability flags       | `toHistoryItem()` sets `canResume` for status `active` or `pending`, `canRate` for `rating_pending`, and `canBookAgain` for completed/rating/cancelled emergencies.  | These flags are presentation affordances; resume still needs selected-history-to-active-request matching.                                           |
| Detail model collapsed guard        | `useMapVisitDetailModel()` later gates collapsed resume through `matchesLiveRequest`, comparing history request/id to `activeMapRequest.requestId`.                  | The collapsed detail path refines selector output with live tracking truth, but primary detail CTA still bypasses that guard.                       |

Fix shape: treat history selector `primaryAction` as an affordance proposal, not final action truth. Every resume path must re-check active request identity; every rating path must treat `rating_pending` as terminal/recovery state rather than active tracking.

### History Detail Resume Identity Evidence

| Surface / path                | Current implementation evidence                                                                                                                                                             | Audit interpretation                                                                                                                                            |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Active request key set        | `useMapShell()` exposes `activeHistoryRequestKeys` from `activeMapRequest.requestId`, `activeMapRequest.id`, `activeMapRequest.record.displayId`, active trip display ids, and pending ids. | The set is useful for history-row matching, but it omits some raw canonical ids such as `activeAmbulanceTrip.id`, `activeAmbulanceTrip.requestId`, and bed ids. |
| History row direct selection  | `handleSelectHistoryItem()` compares selected `requestId`, `displayId`, and `id` against `activeHistoryRequestKeys` before calling `openTracking()`.                                        | The recents/history row path is selected-item-aware, though only as complete as the active key set.                                                             |
| Visit detail primary CTA      | `resolveDetailsPrimaryAction()` returns Resume when `historyItem.primaryAction` is `resume_tracking` or `resume_request` and `onResume` exists.                                             | The primary detail CTA is action-label driven; it does not receive or enforce active request identity.                                                          |
| Visit detail resume handler   | `handleResumeHistoryRequest()` checks only `hasActiveTrip`, then closes the detail surface and calls `openTracking()`.                                                                      | A selected inactive history detail can still resume whichever active trip exists globally.                                                                      |
| Route-opened visit detail     | `useMapRouteHandlers()` opens `mapSheet=visit_detail&visitKey=...` by calling `openHistoryVisitByKey(routeVisitKey, { routeManaged: true })`.                                               | Deep-linked history detail inherits the same global resume handler, so route identity is not enough to protect resume.                                          |
| Collapsed visit-detail action | `useMapVisitDetailModel()` computes `matchesLiveRequest` and gates `collapsedAction` resume on `historyItem.requestId/id` matching `activeMapRequest.requestId`.                            | The collapsed action already models the intended invariant, but the primary CTA is a separate path and can diverge.                                             |
| Stale selected detail cleanup | `closeHistoryVisitDetails()` clears `selectedHistoryVisitKey`; a sheet-phase effect also clears it whenever the phase leaves `VISIT_DETAIL`.                                                | This prevents stale map focus, but it does not prove the selected visit is the active request at resume time.                                                   |
| Map focus from history detail | `historyFocusedHospital` can be derived from selected visit hospital id/name or facility coordinates.                                                                                       | This is valid detail/map context, not active request truth; it must not be used as proof that the visit is resumable.                                           |

Fix shape: expand the active key set to include raw active ambulance/bed canonical ids, derive `selectedHistoryMatchesActiveRequest` for the selected detail item, and use that boolean for both the detail primary CTA and `handleResumeHistoryRequest()` before `openTracking()`.
